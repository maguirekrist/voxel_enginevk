# Structure System Plan

## Goal

Build a moddable, modular, flexible, and expressive structure system for the voxel world that supports:

- world-coordinate placement instead of chunk-owned generation
- deterministic seeded generation
- structures spanning chunk borders
- future expansion to multiple tree types, villages, dungeons, castles, and other authored or procedural content
- eventual mod/data-driven structure definitions

## Current State

The current implementation has the right first step, but it is still not a full structure pipeline:

- structure edits are generated in world coordinates
- each chunk queries overlapping tree candidates
- tree placement is deterministic in world-space cells

What is still missing:

- an explicit placement phase
- cached anchors or instances shared across chunk generation
- a clear separation between placement, resolution, and stamping
- a path for multi-piece structures and moddable templates

## Target Architecture

The structure system should be split into four layers.

### 1. Placement

Responsibility:

- decide where structures want to exist in world coordinates
- evaluate biome, terrain, density, slope, spacing, and seed rules
- emit deterministic `StructureAnchor` records

Examples:

- tree candidate at `(worldX, worldY, worldZ)`
- village center anchor
- dungeon entrance anchor
- castle site anchor

This should be owned by a dedicated `StructurePlacer`.

### 2. Resolution

Responsibility:

- convert an anchor into a resolved structure instance
- choose variant, size, rotation, palette, and child pieces
- stay deterministic from world seed + anchor seed

Examples:

- oak tree vs pine tree
- small dungeon vs large dungeon
- stone castle vs mossy castle

This should be owned by structure generators registered in `StructureRegistry`.

### 3. Stamping

Responsibility:

- turn a resolved structure instance into world-space block edits or piece placements
- support cross-chunk footprints naturally
- avoid direct chunk-local assumptions

This is where voxel output is produced.

### 4. Chunk Application

Responsibility:

- query overlapping structure output for the current chunk
- apply only the edits intersecting that chunk
- stay independent from how placement/resolution were performed

This remains the final filtering step in chunk generation.

## Proposed Core Types

The structure system should evolve toward these core concepts.

### `StructureAnchor`

Purpose:

- compact placement record generated by the placement pass

Suggested fields:

- `StructureType type`
- `glm::ivec3 worldOrigin`
- `uint64_t seed`
- `uint32_t biomeId`
- optional placement metadata such as terrain height or placement tags

### `StructureInstance`

Purpose:

- resolved structure output before voxel stamping

Suggested fields:

- `StructureType type`
- `glm::ivec3 worldOrigin`
- bounds or footprint
- variant id
- rotation
- palette id
- piece list or procedural parameters

### `StructureBlockEdit`

Purpose:

- final voxel modifications in world coordinates

This already exists and should remain part of the pipeline.

### `StructurePlacer`

Purpose:

- generate anchors for a world region
- own placement rules and spacing logic

### `StructureIndex`

Purpose:

- cache anchors and optionally resolved instances by region
- let many chunks query the same placement data without recomputing it independently

Region granularity can start simple, such as:

- one cache entry per `N x N` chunk region
- deterministic regeneration on demand if a region is missing

### `StructureRegistry`

Purpose:

- register structure families and their generators
- own resolution and stamping logic per structure type

## Recommended Execution Model

### Phase 1: Explicit Placement Pass

Add a `StructurePlacer` that can generate anchors for a world region.

Requirements:

- placement uses world coordinates only
- no chunk-local random selection
- deterministic from world seed and region coordinates
- terrain queries use `TerrainGenerator::SampleHeight(...)`

Expected output:

- a collection of `StructureAnchor`s for a region

### Phase 2: Region-Based Structure Index

Add a `StructureIndex` that stores anchors by region.

Requirements:

- chunk generation queries overlapping regions
- anchors are generated once per region instead of re-evaluated ad hoc per chunk
- the index can later support persistence or serialization

This is the step that makes the system clearly become an explicit placement pass rather than an on-demand helper.

### Phase 3: Resolution Layer

Refactor structure generation so `StructureRegistry` resolves anchors into instances.

Requirements:

- tree generators stop doing placement work
- generators receive anchors and produce `StructureInstance` or block edits
- variant selection becomes part of the resolved instance

For trees, this can start simple:

- `Oak`
- `Birch`
- `Pine`

### Phase 4: Multi-Piece Structures

Add support for structures composed of pieces rather than only direct voxel edits.

Requirements:

- piece definitions with transforms
- optional sockets or connection points
- deterministic piece expansion

This is the basis for:

- villages
- dungeons
- castles
- ruins

### Phase 5: Data-Driven Definitions

Add data-driven structure content for moddability.

Suggested moddable inputs:

- palettes
- placement rules
- piece templates
- structure metadata
- variant weights

Code should still own:

- deterministic seed handling
- placement execution
- validation
- chunk overlap queries
- voxel stamping core

## Immediate Refactor Path

When implementation starts, the work should happen in this order.

1. Introduce `StructurePlacer`.
   It should generate anchors for a region in world coordinates.

2. Introduce `StructureRegionKey` and `StructureIndex`.
   Chunks should query overlapping regions instead of generating placement candidates directly.

3. Change chunk generation to consume anchors from the index.
   `ChunkData::generate()` should no longer know how structures are placed.

4. Keep trees as the first structure family.
   The first migrated generator should stay intentionally simple.

5. Add at least one second tree variant.
   This validates that resolution is properly separated from placement.

6. Add bounds/footprint metadata to resolved structures.
   This prepares the system for large structures and region queries.

## Tree-Specific First Implementation

The next tree-focused implementation should use this model:

- `StructurePlacer` places tree anchors using spacing and terrain rules
- `TreeGenerator` resolves an anchor into a tree variant and shape parameters
- the tree generator emits world-space edits
- `ChunkData` only applies overlapping edits

Tree placement rules should support:

- biome gating later
- elevation checks
- water avoidance
- slope checks later
- deterministic spacing through placement cells, blue noise, or similar distribution

## Village / Dungeon / Castle Expansion Path

These should not be implemented as giant one-off block stamping functions.

Instead:

- placement decides where a site exists
- resolution chooses the structure layout
- piece-based generation assembles rooms, roads, towers, walls, and houses
- stamping converts pieces into voxel output

Recommended future model:

- villages use centers plus road/house pieces
- dungeons use room/corridor graphs
- castles use keeps, towers, gates, wall segments, and courtyards

## Modding Direction

For moddability, the engine should eventually separate:

- engine-owned execution logic
- data-owned content definitions

Good candidates for data files:

- structure templates
- palettes
- piece sockets
- spawn weights
- biome restrictions
- variant metadata

Good candidates to keep in code:

- deterministic placement framework
- region caching/indexing
- terrain and overlap queries
- core stamping rules

## Success Criteria

The structure system should be considered on the right path when:

- placement is explicit and region-based
- chunks no longer decide structure placement themselves
- anchors can be reused by multiple chunk queries
- trees can span chunk borders naturally
- adding a new tree type does not require editing chunk generation code
- future villages/dungeons/castles can be introduced as new generators instead of bespoke worldgen hacks

## First Implementation Boundary

When this plan is implemented, keep scope controlled:

- do not build full village/dungeon logic yet
- do not add save persistence yet
- do not add a full mod loader yet

The first milestone should simply prove:

- explicit structure placement pass
- region cache/index
- tree anchors resolved from that pass
- chunk application from shared world-space structure data
