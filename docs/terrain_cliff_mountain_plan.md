# Terrain Cliff, Shelf, And Mountain Volume Plan

> Superseded as the primary architecture document by [world_generation_architecture_plan.md](C:/Users/magui/source/repos/voxel_enginevk/docs/world_generation_architecture_plan.md). This file is now a narrower terrain-feature-focused reference, not the authoritative full world-generation plan.

## Purpose

Define a terrain-generation architecture that can produce:

- bubbly mountain masses
- shelf plateaus
- walk-under overhangs
- future volumetric terrain artifacts

while preserving the project preference for layered world generation and debuggable strategy stages.

## Correction To The Earlier Plan

The previous plan over-focused on heightfield cliff shaping.

That can create:

- steeper escarpments
- terraced ridges
- sharper surface transitions

but it cannot create a true overhang, because a pure heightfield still only stores one solid surface per `(x, z)` column.

The CubeWorld-like target in the reference image clearly includes terrain where one column can have:

- solid below
- air in the middle
- solid again above

That is a volumetric terrain feature, not just a steeper heightmap cliff.

## Core Design Decision

The terrain generator should keep two distinct layer families.

### 1. Heightfield Layers

These remain responsible for macro terrain:

- continents
- valleys
- ridges
- rivers
- biome classification
- surface material hints

These layers continue to operate on `ChunkTerrainData` and `TerrainColumnSample`.

### 2. Voxel Terrain Feature Layers

These operate on actual chunk blocks after the base terrain has been filled.

These layers are responsible for:

- additive mountain bubble masses
- shelf plateaus
- overhangs
- future arches, caves, and other volumetric artifacts

These layers should be composed exactly like the heightfield layers: as ordered strategy stages in a template-method generation flow.

## Target Generation Flow

The intended runtime order is:

1. `BaseTerrainLayer`
2. biome / surface-oriented heightfield layers
3. fill base solid terrain blocks from the final heightfield
4. run voxel terrain feature layers
5. refresh final exposed surface materials
6. apply structures
7. place decorations

This keeps generation understandable and debuggable:

- heightfield stages explain macro shape
- voxel feature stages explain overhang-capable additions
- structures remain a later, separate concern

## Terminology

To reduce confusion, distinguish these concepts.

### Heightfield Escarpment

A steep surface drop generated by changing `surfaceHeight`.

This is still useful, but it is not a true overhang.

### Shelf / Overhang Feature

A volumetric terrain mass attached to a slope or plateau edge, often with empty space underneath.

This is the correct concept for the CubeWorld-like foreground landform.

### Mountain Bubble Feature

A rounded or blobby volumetric mass that merges into terrain and makes mountains feel like they are swelling upward from below.

This is the correct concept for the bubbly mountain ranges in the reference image.

## Design Principles

- Keep layer composition explicit and ordered.
- Do not hide volumetric terrain behavior inside ad hoc chunk code.
- Keep each feature family in its own strategy layer.
- Use world-space deterministic placement so features cross chunk boundaries cleanly.
- Treat volumetric terrain features as first-class terrain, not as structures.
- Keep caves as a later voxel feature family, not the first target.

## Heightfield Layer Responsibilities

Heightfield layers should continue to own:

- 2D noise-driven base landmass shaping
- biome classification
- river/ocean/shore flattening
- material hints
- optional escarpment-style sharpening
- diagnostic values like slope and ridge masks

These layers are still valuable because they provide the coarse terrain scaffold the voxel feature layers attach to.

## Voxel Terrain Feature Layer Responsibilities

Voxel terrain feature layers should own:

- additive 3D terrain masses
- volumetric shelves
- true overhangs
- volumetric breakup
- later subtractive cave and arch systems

These layers should operate on the chunk's voxel data after the base heightfield has already been rasterized into blocks.

## First Voxel Feature Families

## Mountain Bubble Layer

Purpose:

- make mountain ranges feel spherical, swollen, and mass-driven
- add large rounded volume above and around the base terrain scaffold

Key idea:

- place deterministic bubble volumes in mountain regions
- merge them with the existing terrain
- modulate them with low-frequency 3D noise so they do not look like perfect ellipsoids

This should produce the "terrain bubbling up from the ground" feeling described by the user.

## Shelf Overhang Layer

Purpose:

- create walk-under plateau shelves and ridge outcrops
- attach volumetric caps to steep slopes

Key idea:

- find steep mountainous anchor regions
- extend a cap volume outward in the downslope direction
- keep the top relatively walkable
- shape the underside with curved/noisy volume so it reads as terrain, not as a hard slab

This produces overhang-capable landforms instead of just steep surface walls.

## Why These Should Be Separate Layers

Because they solve different problems.

- mountain bubbles create broad mass
- shelf layers create lateral extension and walk-under space

They should be independently:

- enabled
- tuned
- debugged
- reordered if needed

That is exactly the kind of layered strategy architecture the user asked for.

## Surface Material Refresh

Once volumetric terrain features add solid blocks above the original heightfield, the old "top block at `surfaceHeight`" assumption is no longer enough.

So the generation flow must include a surface-material refresh pass that:

- finds the highest exposed solid in each column
- repaints the top surface based on biome/material rules
- gives shelf tops grass or sand when appropriate
- leaves undersides and deep interior as rock

Without this, volumetric shelves would be generated as raw stone everywhere.

## Debuggability Requirements

The architecture should make it easy to inspect:

- macro heightfield result
- escarpment mask result
- mountain bubble placement
- shelf placement
- final solid voxel outcome

In practice this means:

- keep feature families in their own layer classes
- give each layer a stable name
- make settings explicit and serializable
- keep chunk generation flow visibly staged

## Suggested Settings Shape

The first volumetric settings group should support:

- enable/disable entire volume feature pipeline
- enable/disable bubble layers
- enable/disable shelf layers
- placement cell size
- minimum elevation for features
- spawn chance
- radius / height / thickness ranges
- 3D noise frequency and strength
- slope requirement for shelf attachment

This is enough for the first pass without over-designing the tuning surface.

## Future Extension Path

Once the first volumetric layers work, later terrain feature layers can include:

- cave carve layers
- arch layers
- river undercut layers
- erosion crack layers
- biome-specific terrain masses

The key is that each one still plugs into the same voxel terrain feature layer pipeline.

## Recommended Implementation Phases

## Phase 1

Introduce a separate voxel terrain feature layer abstraction and move chunk generation onto an explicit staged flow:

- base fill
- voxel terrain feature layers
- surface refresh

## Phase 2

Add `MountainBubbleFeatureLayer`.

## Phase 3

Add `ShelfOverhangFeatureLayer`.

## Phase 4

Tune surface material refresh so shelf tops and mountain caps read correctly.

## Phase 5

Add visualization/debug outputs for volumetric feature placement and affected columns.

## Phase 6

Later add subtractive cave/arch layers.

## Success Criteria

This plan succeeds when:

- some terrain columns contain multiple solid spans
- mountain regions visibly bulge upward as volume, not just as perturbed surface height
- shelves can create walk-under space
- the generation architecture remains layered and debuggable
- future volumetric terrain artifacts have a clear place to plug in
